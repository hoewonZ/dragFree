# 版本号策略

本项目采用语义化版本（Semantic Versioning）：`MAJOR.MINOR.PATCH`。  
版本号更新采用“提交后确认”流程：先完成代码提交，再由 Agent 询问是否更新版本号。

## 版本格式

- 正式版本：`X.Y.Z`（例如 `0.2.1`）
- 预发布版本（可选）：`X.Y.Z-beta.N`、`X.Y.Z-rc.N`

## 升级规则

### PATCH（补丁，`X.Y.Z+1`）

用于**向后兼容**的修复与小优化：

- Bug 修复（拖拽/投放、命中区域、编辑器行为、配置页问题）
- 不改变外部行为的小型重构
- 样式与性能调优（不新增能力）

### MINOR（次版本，`X.Y+1.0`）

用于**向后兼容**的新功能：

- 新增用户可见功能（新按钮、新开关、新交互能力）
- 新增配置字段，且具备默认值与归一化/兼容逻辑
- 新增可选行为，不破坏旧配置可用性

### MAJOR（主版本，`X+1.0.0`）

用于**不兼容变更**：

- 删除/重命名配置键，且没有兼容回退
- 主进程、预加载、渲染进程之间的 IPC 协议破坏性变更
- 现有用户流程必须迁移后才能继续使用

## 提交后版本更新流程（必须执行）

当 Agent 完成一次代码提交后，必须执行以下交互流程：

1. 向用户询问“是否更新版本号”。
2. 同时展示升级依据（变更类型、是否新增功能、是否破坏兼容）并给出建议升级级别（PATCH / MINOR / MAJOR）。
3. 仅当用户明确确认后，才更新 `package.json` 中的 `"version"`。
4. 更新 `COMMIT_HISTORY.md`，记录版本升级原因与主要改动摘要。
5. 完成版本号提交（建议使用独立提交，便于追踪）。

## Conventional Commit 前缀与版本映射（建议）

- `fix:` -> 通常对应 `PATCH`
- `feat:` -> 至少对应 `MINOR`
- `perf:`、`refactor:`、`style:` -> 通常对应 `PATCH`（若带来明显新能力则可升 `MINOR`）
- 任何不兼容改动 -> 无论前缀，统一升 `MAJOR`

## 示例

- `0.1.0` -> `0.1.1`：修复拖拽文字追加问题
- `0.1.1` -> `0.2.0`：新增文本 tab 与系统配置开关
- `0.2.3` -> `1.0.0`：移除旧配置结构且无兼容迁移

## 交互示例（提交后）

- Agent：是否需要更新版本号？  
  依据：本次新增系统设置项与拖拽文本行为，属于向后兼容的新功能，建议 `MINOR`（例如 `0.2.0 -> 0.3.0`）。
- 用户确认后：Agent 再执行版本文件更新与版本提交。
